# Redaction Guide

Aweb Agent Receipts are useful only if they preserve accountability without turning
agent systems into surveillance-grade raw logs.

## Public By Default

The following fields are generally safe to expose in reviewer-facing examples:

- receipt id,
- example/demo marker,
- scoped authority mode,
- allowed and denied action classes,
- workflow intent at a high level,
- provider or capability class,
- execution status,
- timing,
- idempotency key,
- cost class,
- failure/retry/recovery state,
- digest-grade artifact references,
- redaction reason,
- human-review state.

## Private By Default

The following fields must stay private or be represented only by a digest, placeholder,
or redaction reason:

- raw prompts,
- private user payloads,
- private email threads,
- private business records,
- source code and private repository metadata,
- security payloads and exploit details,
- infrastructure topology,
- deploy commands and environment IDs,
- invoice, tax, vendor, and bank details,
- secrets and API credentials,
- provider tokens,
- wallet secrets,
- viewing keys,
- seed phrases,
- private keys,
- counterparty identities,
- private account data,
- payment memos,
- full prepared transactions,
- private operator notes,
- customer context,
- unsafe automation details.

## Digest-Only Boundary

When a reviewer needs evidence that private data existed without seeing it, the receipt
should use digest-grade references such as `sha256:example-digest`, artifact handles, or
policy references. Public examples should state that these are placeholders unless they
are generated by a real test fixture.

## Review Rule

If publishing a field would make an agent action easier to abuse, deanonymize a user,
expose a private payload, reveal business-sensitive context, leak source/security/
finance data, or imply authority the agent did not have, redact it.
